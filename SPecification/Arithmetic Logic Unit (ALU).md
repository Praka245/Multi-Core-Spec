# Module 9 Specification

## Arithmetic Logic Unit (ALU)

**RTL File:** `alu.v`

**Category:** (A) Datapath Block

---

# 1. Purpose

The **Arithmetic Logic Unit (ALU)** performs all arithmetic, logical, comparison, and address calculation operations required by the RV32I Base Integer ISA.

The ALU receives two 32-bit operands and an ALU control code, performs the selected operation, and produces a 32-bit result along with status flags.

The ALU is reused during different stages of instruction execution, making it one of the most heavily utilized hardware blocks in the Multi-Cycle processor.

---

# 2. Function

The ALU performs the following functions:

- Integer addition
- Integer subtraction
- Bitwise AND
- Bitwise OR
- Bitwise XOR
- Set Less Than (signed)
- Set Less Than Unsigned
- Logical Left Shift
- Logical Right Shift
- Arithmetic Right Shift
- Branch comparison
- Memory address calculation
- Branch target address calculation
- Program Counter increment (PC + 4)

---

# 3. Interface Specification

## Inputs

| Signal | Width | Direction | Description |
|--------|------:|-----------|-------------|
| `operand_a` | 32 | Input | First ALU operand |
| `operand_b` | 32 | Input | Second ALU operand |
| `alu_control` | 4 | Input | ALU operation selector |

---

## Outputs

| Signal | Width | Direction | Description |
|--------|------:|-----------|-------------|
| `alu_result` | 32 | Output | Result of the selected ALU operation |
| `zero` | 1 | Output | Asserted when ALU result equals zero |
| `negative` | 1 | Output | Asserted when result is negative |
| `carry` | 1 | Output | Carry-out from addition/subtraction |
| `overflow` | 1 | Output | Signed arithmetic overflow |

---

# 4. Operand Specification

| Operand | Source |
|----------|--------|
| Operand A | ALUSrcA MUX |
| Operand B | ALUSrcB MUX |

The ALU never directly receives data from the Register File or Program Counter.

All operands are selected through multiplexers.

# 5. Supported Operations

| ALU Control | Operation | Description |
|-------------|-----------|-------------|
| 0000 | ADD | Addition |
| 0001 | SUB | Subtraction |
| 0010 | AND | Bitwise AND |
| 0011 | OR | Bitwise OR |
| 0100 | XOR | Bitwise XOR |
| 0101 | SLT | Signed Less Than |
| 0110 | SLTU | Unsigned Less Than |
| 0111 | SLL | Logical Left Shift |
| 1000 | SRL | Logical Right Shift |
| 1001 | SRA | Arithmetic Right Shift |

> **Note:** The exact encoding of `alu_control` will be generated by the **ALU Control Module** (`alu_control.v`) and can be modified if required.

# 11. Design Decisions

## Why is one ALU sufficient?

A Multi-Cycle processor executes only one operation per clock cycle.

Therefore, the same ALU can be reused for:

- PC increment
- Branch calculation
- Address calculation
- Arithmetic
- Logical operations

This significantly reduces hardware compared to implementing multiple arithmetic units.

---

## Why is the ALU combinational?

The ALU performs only combinational logic.

The computed result is stored in the **ALUOut Register** on the next clock edge.

Keeping the ALU combinational minimizes latency and simplifies timing.

---

## Why generate status flags?

Although only the **Zero flag** is required for the initial RV32I Multi-Cycle implementation, generating additional flags (`Negative`, `Carry`, `Overflow`) improves:

- Debugging
- Verification
- Future ISA extensions
- Scalability

---

# 12. Design Assumptions

- RV32I Base ISA
- 32-bit datapath
- Two 32-bit operands
- Combinational ALU
- One operation performed at a time

---

# 13. Future Scalability
